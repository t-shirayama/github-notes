# コンフリクト解消

mergeやrebaseで同じ箇所が変更されていると、Gitが自動で判断できずコンフリクトすることがあります。

## 実際に試す

コンフリクトは、一度わざと起こしてみると流れを理解しやすいです。この備忘録リポジトリ本体ではなく、練習用リポジトリで [conflict を試す](./ハンズオン/conflict.md) を実行します。

## いつ起きるか

`注意`

### いつ使う

次のような操作中に `CONFLICT` と表示されたとき。

### コマンド例

```sh
git merge main
git merge origin/main
git rebase origin/main
git pull
```

### 注意点

コンフリクトはエラーというより、どちらの変更を残すか人が判断する必要がある状態です。焦って `reset --hard` せず、まず `git status` で状況を確認します。

### 戻し方があるか

merge中なら `git merge --abort`、rebase中なら `git rebase --abort` で中止できます。

## まず確認するコマンド

`安全`

### いつ使う

どのファイルがコンフリクトしているか、merge中かrebase中かを確認したいとき。

### コマンド例

```sh
git status
```

### 注意点

`both modified` や `unmerged paths` と表示されているファイルが、解消対象です。Gitが次に必要な操作も表示してくれるので、まず出力を読みます。

### 戻し方があるか

確認だけなので戻し方は不要です。

## 解消の流れ

`注意`

### いつ使う

コンフリクトしたファイルを手動で直して、mergeやrebaseを続けたいとき。

### コマンド例

```sh
# 1. 状態を確認
git status

# 2. コンフリクトしたファイルを開いて修正
# <<<<<<<, =======, >>>>>>> を消して、残したい内容に整える

# 3. 修正したファイルをステージ
git add <file>

# 4. merge中ならcommit
git commit

# 5. rebase中なら続行
git rebase --continue
```

### 注意点

コンフリクトしたファイルには、次のようなマーカーが入ります。

```txt
<<<<<<< HEAD
今いるブランチ側の変更
=======
取り込み元の変更
>>>>>>> origin/main
```

`<<<<<<<`, `=======`, `>>>>>>>` は最終的にすべて削除します。どちらか片方を残すだけでなく、両方の意図を合わせた内容に直すこともあります。

merge中は、すべてのコンフリクトを解消して `git add` したあとに `git commit` します。rebase中は、`git add` のあとに `git rebase --continue` します。

### 戻し方があるか

途中でやめたい場合は、merge中なら `git merge --abort`、rebase中なら `git rebase --abort` を使います。

## 中止する

`注意`

### いつ使う

コンフリクト解消をやめて、mergeやrebaseを始める前の状態に戻したいとき。

### コマンド例

```sh
# merge中の中止
git merge --abort

# rebase中の中止
git rebase --abort
```

### 注意点

中止する前に `git status` で、今がmerge中なのかrebase中なのか確認します。

### 戻し方があるか

中止後にもう一度取り込みたい場合は、あらためて `git merge` または `git rebase` を実行します。
