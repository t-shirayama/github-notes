# 取り消し操作

変更やcommitを取り消したいときの判断メモです。push前とpush後で、安全な選択肢が変わります。

## 判断表

| 状況 | まず使うコマンド | 危険度 | 考え方 |
| ---- | ---------------- | ------ | ------ |
| ファイルの作業中の変更を捨てたい | `git restore <file>` | `注意` | 未commitの変更が消える |
| 作業中の修正をまとめてリセットしたい | `git restore .` | `注意` | 追跡済みファイルの未commit変更が消える |
| ステージ済みも含めて完全に戻したい | `git reset --hard HEAD` | `危険` | 未commit変更がまとめて消える |
| ステージを外したい | `git restore --staged <file>` | `安全` | 変更内容は残る |
| 直前のcommitメッセージを直したい | `git commit --amend` | `注意` | push前だけにする |
| push前のcommitを取り消したい | `git reset` | `注意` | 履歴を書き換えるがローカルだけなら扱いやすい |
| push済みの修正を安全に取り消したい | `git revert <commit>` | `安全` | 取り消しcommitを追加する |
| push済みの履歴を消したい | `git reset --hard` + `git push --force-with-lease` | `危険` | 共有ブランチでは原則避ける |

## 作業中の変更を戻す

`注意`

### いつ使う

まだcommitしていないファイルの変更を捨てたいとき。

### コマンド例

```sh
git restore <file>
```

### 注意点

未commitの変更が消えます。実行前に必ず `git diff <file>` で内容を確認します。

### 戻し方があるか

commitしていない変更は、基本的に戻せません。不安な場合は先にstashします。

```sh
git stash -u
```

## 作業中の修正を一度リセットする

`注意`

### いつ使う

作業中の修正をいったん捨てて、最後のcommitの状態に戻したいとき。

### コマンド例

```sh
# まず消える内容を確認
git status
git diff

# 追跡済みファイルの未commit変更を全部捨てる
git restore .

# ステージ済みの変更も含めてHEADの状態へ戻す
git reset --hard HEAD
```

追跡していない新規ファイルも削除したい場合:

```sh
# 削除対象を確認するだけ
git clean -nd

# 確認した対象を実際に削除
git clean -fd
```

### 注意点

`git restore .` は、Gitが追跡しているファイルの未commit変更を捨てます。新規作成した未追跡ファイルは残ります。

`git reset --hard HEAD` は、ステージ済み・未ステージの変更をまとめて最後のcommitの状態へ戻します。未commitの修正は戻せないことが多いので、迷ったら先にstashします。

`git clean -fd` はGit管理外のファイルを削除します。実行前に必ず `git clean -nd` で削除対象を確認します。

### 戻し方があるか

未commitの修正は基本的に戻せません。不安な場合は、先に退避します。

```sh
git stash -u
```

## ステージを外す

`安全`

### いつ使う

`git add` したけれど、commitには含めたくない変更があるとき。

### コマンド例

```sh
git restore --staged <file>
```

### 注意点

ステージから外すだけで、ファイルの変更内容は残ります。

### 戻し方があるか

もう一度 `git add <file>` すればステージできます。

## 直前のcommitを修正する

`注意`

### いつ使う

直前のcommitにファイルを足し忘れた、またはcommitメッセージを直したいとき。

### コマンド例

```sh
# ファイルを追加してから直前のcommitに含める
git add <file>
git commit --amend

# メッセージだけを直す
git commit --amend -m "fix: correct validation message"
```

### 注意点

`--amend` は直前のcommitを作り直します。push済みのcommitには原則使わず、push後は `git revert` を使います。

### 戻し方があるか

`git reflog` でamend前のcommitを探して戻せます。

## push前のコミットを取り消す

`注意`

### いつ使う

まだpushしていないcommitを取り消したいとき。

### コマンド例

```sh
# commitだけを取り消し、変更はステージ済みのまま残す
git reset --soft HEAD~1

# commitを取り消し、変更は作業ツリーに残す
git reset --mixed HEAD~1

# commitも変更も消す
git reset --hard HEAD~1
```

### 注意点

`--hard` は作業内容を失う可能性があります。普段は `--soft` か `--mixed` から考えます。

### 戻し方があるか

`git reflog` でreset前のcommitを探して戻せます。ただし、未commitの変更を `--hard` で消した場合は戻せないことがあります。

## push済みの修正を安全に取り消す

`安全`

### いつ使う

すでにpushしたcommitの変更を取り消したいが、履歴は残したいとき。共有ブランチではこの方法が基本です。

### コマンド例

```sh
git revert <commit>
git push origin <branch>
```

複数commitをまとめてrevert commitにしたい場合:

```sh
git revert --no-commit <oldest-commit>^..<newest-commit>
git commit -m "revert: remove accidental changes"
git push origin <branch>
```

### 注意点

`revert` は対象commitを消すのではなく、反対の変更を入れる新しいcommitを作ります。履歴が残るので、チーム開発で安全に扱いやすいです。

### 戻し方があるか

revert自体を取り消したい場合は、そのrevert commitをさらに `git revert <revert-commit>` します。

## merge commitを取り消す

`注意`

### いつ使う

merge commitを履歴に残したまま取り消したいとき。

### コマンド例

```sh
git revert -m 1 <merge-commit>
```

### 注意点

`-m 1` はmainlineの親を指定します。どちらの親を基準にするか分からない場合は、先に `git show <merge-commit>` で確認します。

### 戻し方があるか

通常のrevertと同じく、revert commitをさらにrevertできます。
