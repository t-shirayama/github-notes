# 履歴の書き換え

履歴を書き換える操作は強力ですが、共有ブランチでは事故につながります。実行前に影響範囲を確認し、必要ならチームへ共有します。

このページでは、`reset --hard`、`rebase -i`、`push --force-with-lease` など、履歴そのものを変える操作を詳しく扱います。安全に取り消すだけなら、まず [取り消し操作](./取り消し操作.md#push済みの修正を安全に取り消す) の `git revert` を使います。

## push済みの修正を履歴ごと削除する

`危険`

### いつ使う

誤ってpushしたcommitを、履歴からも消したいとき。機密情報を含むcommitなど、履歴に残すこと自体が問題になる場合に検討します。

### コマンド例

```sh
# 戻したいcommitを確認
git log --oneline

# 指定commitまでブランチを戻す
git reset --hard <戻したいcommit>

# リモートを上書きする
git push --force-with-lease origin <branch>
```

### 注意点

共有ブランチでは原則避けます。`--force-with-lease` は、自分が最後に取得したリモート状態から他人の更新が進んでいない場合にだけ上書きするため、単純な `--force` より安全です。

ただし、履歴を書き換えること自体の危険は残ります。すでに他の人がpullしている場合は、必ずチームで対応方針を合わせます。

### 戻し方があるか

ローカルに履歴が残っていれば `git reflog` から戻せます。リモートや他メンバーの環境に影響するため、戻す場合もチームへ共有します。

## push済みの修正を履歴を残して取り消す場合

通常のチーム開発では、履歴を書き換えずに `git revert` で取り消す方法を優先します。手順は [取り消し操作](./取り消し操作.md#push済みの修正を安全に取り消す) にまとめています。

## reset --hard でローカル履歴を戻す

`危険`

### いつ使う

ローカルブランチを特定のcommitの状態に完全に戻したいとき。

### コマンド例

```sh
git reset --hard <commit>
```

### 注意点

commitだけでなく、作業ツリーの変更も消える可能性があります。実行前に `git status` と `git log --oneline` を確認します。

### 戻し方があるか

`git reflog` でreset前のcommitを探して戻せます。

```sh
git reflog
git reset --hard <戻したいcommit>
```

## rebase -i でcommit履歴を整理する

`危険`

### いつ使う

push前の作業ブランチで、commitをまとめる、並べ替える、メッセージを直すなど履歴を整理したいとき。

### コマンド例

```sh
git rebase -i HEAD~3
```

よく使う操作:

| 操作 | 意味 |
| ---- | ---- |
| `pick` | そのcommitを使う |
| `reword` | commitメッセージを直す |
| `squash` | 前のcommitにまとめる |
| `drop` | commitを削除する |

### 注意点

履歴を書き換えるため、push済み・共有済みのcommitには原則使いません。

### 戻し方があるか

rebase中なら次で中止できます。

```sh
git rebase --abort
```

rebase後なら `git reflog` でrebase前のcommitを探します。

## 過去の履歴をすべて削除する

`危険`

### いつ使う

個人開発で履歴を作り直したいときなど、かなり限定的な場面です。チーム開発や共有リポジトリでは原則使いません。

### コマンド例

```sh
git switch --orphan tmp
git add -A
git commit -m "Initial commit"
git branch -M main
git push --force-with-lease origin main
```

### 注意点

全履歴を書き換えます。既存の履歴を参照しているPR、タグ、他メンバーのローカル環境に大きな影響があります。

既存メモの `git checkout --orphan tmp` と同じ目的ですが、ブランチ操作は `git switch --orphan` を優先して書いています。

### 戻し方があるか

ローカルやリモートに古いcommitが残っていれば復元できる可能性があります。実行前にバックアップブランチを作る方が安全です。

```sh
git branch backup-before-rewrite
```

## force push の使い分け

`危険`

### いつ使う

rebaseやresetで書き換えたローカル履歴をリモートへ反映したいとき。

### コマンド例

```sh
git push --force-with-lease origin <branch>
```

### 注意点

`git push --force` は、他人のpushを上書きする危険があります。使う必要がある場合でも、基本は `--force-with-lease` を選びます。

### 戻し方があるか

上書き前のcommitが分かれば戻せます。分からない場合は自分や他メンバーの `git reflog` を探します。
