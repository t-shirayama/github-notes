# 日常操作

普段の開発でよく使う、状態確認・差分確認・コミット・履歴確認のコマンドです。

## 状態を確認する

`安全`

### いつ使う

作業ツリーに変更があるか、どのファイルがステージ済みか、今いるブランチがどこかを確認したいとき。

### コマンド例

```sh
git status
git status --short
```

### 注意点

`git status` は状態を見るだけなので安全です。作業開始前、commit前、push前に見る癖をつけると事故が減ります。

### 戻し方があるか

状態確認だけなので戻し方は不要です。

## 差分を確認する

`安全`

### いつ使う

変更内容を確認したいとき。レビュー前やcommit前に必ず見ると安心です。

### コマンド例

```sh
# まだステージしていない差分
git diff

# ステージ済みの差分
git diff --staged

# mainとの差分
git diff main...HEAD
```

### 注意点

`main..HEAD` と `main...HEAD` は意味が違います。PRの差分確認では、分岐元からの差分を見やすい `main...HEAD` を使うことが多いです。

### 戻し方があるか

差分確認だけなので戻し方は不要です。

## 変更を選んでステージする

`安全`

### いつ使う

1つのファイルに複数の変更があり、commitを分けたいとき。

### コマンド例

```sh
git add -p
```

よく使う選択肢:

| 入力 | 意味 |
| ---- | ---- |
| `y` | この変更をステージする |
| `n` | この変更をステージしない |
| `s` | 変更をさらに細かく分割する |
| `q` | 終了する |

### 注意点

commitの粒度を整えたいときに便利です。迷ったら `n` か `q` で止められます。

### 戻し方があるか

ステージを外すだけなら次を使います。

```sh
git restore --staged <file>
```

## コミットする

`安全`

### いつ使う

ステージした変更を履歴として保存したいとき。

### コマンド例

```sh
git commit -m "feat: add user profile"
```

### 注意点

commit前に `git diff --staged` で、意図した変更だけが入っているか確認します。

### 戻し方があるか

push前なら `git reset` で戻せます。push後なら原則 `git revert` を使います。

## origin とは

`安全`

### いつ使う

`git fetch origin` や `git push origin main` の `origin` が何を指しているか確認したいとき。

### コマンド例

```sh
# 登録されているリモート名とURLを確認
git remote -v

# originから最新情報を取得
git fetch origin

# originのmainブランチへpush
git push origin main
```

### 注意点

`origin` はリモートリポジトリの名前です。多くの場合、clone元のリポジトリに自動で付く標準的な名前です。

`origin/main` は「originというリモートにあるmainブランチを、最後に取得した時点でローカルに記録したもの」です。自分のローカルブランチ `main` とは別物です。

| 表記 | 意味 |
| ---- | ---- |
| `origin` | リモートリポジトリ名 |
| `main` | ローカルブランチ名 |
| `origin/main` | リモート追跡ブランチ |
| `git push origin main` | ローカルの `main` を `origin` の `main` へ送る |
| `git fetch origin` | `origin` の最新情報を取得する |

### 戻し方があるか

確認やfetchだけなら戻し方は不要です。リモートURLを変更した場合は、次のように再設定できます。

```sh
git remote set-url origin <url>
```

## リモートの変更を取得する

`安全`

### いつ使う

リモートの最新状態を確認したいとき。作業中の変更に混ぜず、まず情報だけ取りたい場合に使います。

### コマンド例

```sh
git fetch origin
git fetch --prune
```

### 注意点

`fetch` はリモート追跡ブランチを更新しますが、現在の作業ブランチには直接反映しません。消えたリモートブランチの整理には `--prune` が便利です。

### 戻し方があるか

通常は戻し方を意識する必要はありません。

## リモートの変更を取り込む

`注意`

### いつ使う

現在のブランチに、対応するリモートブランチの変更を取り込みたいとき。

### コマンド例

```sh
git pull origin main
```

### 注意点

`git pull` は基本的に `fetch` と統合操作をまとめて行います。コンフリクトが起きることがあるので、作業中の変更がある場合は先にcommitするかstashします。

### 戻し方があるか

merge commitが作られた場合は `git revert -m 1 <merge-commit>` で戻せます。作業中に不安になったら、まず `git status` を確認します。

## 履歴を見る

`安全`

### いつ使う

commit履歴、ブランチの分岐、最新commitの流れを確認したいとき。

### コマンド例

```sh
git log --oneline
git log --oneline --graph --decorate --all
```

### 注意点

`--graph --decorate --all` を使うと、ブランチの関係が見やすくなります。

### 戻し方があるか

履歴確認だけなので戻し方は不要です。

## 特定のコミットを見る

`安全`

### いつ使う

あるcommitで何が変更されたか、commitメッセージや差分を確認したいとき。

### コマンド例

```sh
git show <commit>
git show --stat <commit>
```

### 注意点

`--stat` を付けると変更ファイルと行数の概要だけを見られます。大きいcommitの確認に便利です。

### 戻し方があるか

確認だけなので戻し方は不要です。
