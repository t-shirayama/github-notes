# Issue運用

Issueは、やりたいこと、困っていること、調査したいことを残す場所です。あとから見ても目的と完了条件が分かるように書くと、PRや検索とつながりやすくなります。

## Issueに書くこと

- 背景。
- やりたいこと。
- 完了条件。
- 関連PR。
- 参考リンク。

## テンプレート例

```md
## 背景

- 

## やりたいこと

- 

## 完了条件

- [ ] 

## 関連

- 
```

## ラベル例

| Label | 意味 |
| ----- | ---- |
| `bug` | 不具合 |
| `feature` | 機能追加 |
| `docs` | ドキュメント |
| `refactor` | 振る舞いを変えない整理 |
| `question` | 質問や調査 |
| `blocked` | 他の作業待ち |

Labelsは増やしすぎると運用しにくくなります。最初は種類と状態が分かる程度で十分です。

## 完了条件

- 実装が完了している。
- テストが通っている。
- PRがmergeされている。
- 必要なドキュメントが更新されている。

## PRとのつなげ方

PR本文に `Closes #123` や `Fixes #123` と書くと、PRがmergeされたときにIssueを自動で閉じられます。

```md
Closes #123
```

関連はあるが自動closeしたくない場合は、`Related #123` のように書きます。

## 探し方

Issueが増えてきたら、[GitHubでよく使う検索](./GitHubでよく使う検索.md) の `is:issue`、`label:`、`milestone:` を使って探します。
