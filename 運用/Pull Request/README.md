# Pull Request運用

Pull Request は、変更内容を共有し、レビューし、mainへ安全に取り込むための入口です。小さく出し、目的と確認方法を明確にするとレビューしやすくなります。

## 基本の流れ

1. 最新のmainから作業ブランチを作る。
2. 変更を小さめにcommitする。
3. Draft PRとして早めに出す。
4. CIが通る状態にする。
5. Ready for reviewにしてレビュー依頼する。
6. 指摘対応後、承認と必要な確認を終えてmergeする。

## PRタイトル

タイトルは、何を変えるPRか一目で分かるようにします。

```text
docs: add GitHub Actions notes
fix: handle empty search result
feat: add monthly budget summary
```

迷ったら [コミットメッセージ規約](../コミットメッセージ/README.md) と同じ粒度で書きます。

## PR説明

PR本文には、レビューで見てほしい情報を先に置きます。使い回す形は [PRテンプレート](./テンプレート.md) にまとめています。

```md
## 概要
- 何を変更したか

## 確認したこと
- 実行したテストや確認手順

## 補足
- レビューで見てほしい点
- 今回やらないこと
```

## レビュー依頼

| 状況 | やること |
| ---- | -------- |
| まだ途中 | Draft PRにする |
| 設計だけ相談したい | 相談したい点を本文に書く |
| 実装完了 | Ready for reviewにしてreviewerを指定する |
| 急ぎ | 期限や背景をコメントで補足する |

レビュー依頼前に、差分が大きすぎないか、CIが落ちたままになっていないかを確認します。

レビュー時に見る観点は [レビュー観点](../レビュー観点.md) を参照します。

## Labels と Milestone

Labels はPRの種類や状態を見分けるために使います。

| Label例 | 意味 |
| ------- | ---- |
| `docs` | ドキュメント変更 |
| `bug` | 不具合修正 |
| `feature` | 機能追加 |
| `refactor` | 振る舞いを変えない整理 |
| `blocked` | 他の作業待ち |
| `needs-review` | レビュー待ち |

Milestone は、リリースや作業区切りに紐づくPRをまとめたいときに使います。個人開発では無理に使わず、複数PRを同じ目標にまとめたいときから始めると扱いやすいです。

## merge前チェック

- PRの目的が本文から分かる。
- CIが通っている。
- 不要な差分が混ざっていない。
- レビューコメントに対応済み。
- mainとの差分が古すぎない。
- merge後に消してよいブランチか確認した。

## merge方法の目安

| 方法 | 向いている場面 |
| ---- | -------------- |
| Squash merge | 小さなcommitを1つにまとめてmainを読みやすくしたい |
| Merge commit | 複数commitの流れやPR単位の履歴を残したい |
| Rebase merge | 直線的な履歴を保ちたい |

チームでは、どれを基本にするかを先に決めておくと履歴が読みやすくなります。
