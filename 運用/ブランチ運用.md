# ブランチ運用

ブランチ運用は、mainを安定させながら作業を進めるためのルールです。個人開発でも、mainを常に動く状態にしておくと戻しやすくなります。

## 基本方針

- mainは常に動く状態にする。
- 作業はmainから切ったブランチで行う。
- 変更は小さめのPRに分ける。
- push済みの共有ブランチでは、履歴の書き換えを避ける。
- merge後の作業ブランチは削除する。

## ブランチの種類

| ブランチ | 役割 |
| -------- | ---- |
| `main` | 安定版。直接commitしない |
| `feature/*` | 機能追加やドキュメント追加 |
| `fix/*` | 不具合修正 |
| `hotfix/*` | 急ぎの本番修正 |
| `docs/*` | ドキュメント変更 |

小規模なリポジトリでは、`main` と作業ブランチだけで十分です。`develop` や `release/*` は、リリース管理が必要になってから増やします。

## ブランチ名

ブランチ名は、種類と内容が分かるようにします。

```sh
git switch -c docs/add-github-actions-notes
git switch -c fix/login-error-message
git switch -c feature/monthly-summary
```

日本語より英小文字、数字、ハイフンを使うと、URLやCLIで扱いやすくなります。

## mainの取り込み方

作業ブランチがmainから古くなったときは、mergeかrebaseで取り込みます。

| 方法 | 使いどころ |
| ---- | ---------- |
| `git merge main` | PRレビュー中、履歴を書き換えたくないとき |
| `git rebase origin/main` | push前の個人ブランチをきれいにしたいとき |

詳しい考え方は [ブランチ操作](../コマンド/ブランチ操作.md#バックマージと-rebase-の違い) を参照します。

## push済みブランチの注意

push済みのブランチでrebaseやamendをすると、リモートと履歴がずれます。その場合は `git push --force-with-lease` が必要になることがあります。

共有ブランチでは、他人の作業を壊す可能性があるため慎重に扱います。危険操作の詳細は [履歴の書き換え](../コマンド/履歴の書き換え.md) を確認します。

## ブランチ保護

mainを守りたい場合は、GitHubのBranch protection rulesを使います。

- PR経由のmergeを必須にする。
- CIの成功を必須にする。
- レビュー承認を必須にする。
- force pushを禁止する。
- 削除を禁止する。

最初は「mainへ直接pushしない」「CI成功を確認する」だけでも効果があります。

## 後片付け

merge後は、不要になった作業ブランチを削除します。

```sh
git branch -d <branch>
git push origin --delete <branch>
```

削除してもcommitはmainやPR履歴に残るため、通常は問題ありません。
