# レビュー観点

レビューは、間違い探しだけではなく、変更の意図、影響範囲、保守しやすさを確認するために行います。すべてを毎回細かく見るより、PRの性質に合わせて重点を決めます。

## 仕様

- 要件と実装が一致しているか。
- 想定外の入力や権限不足を考慮しているか。
- エラー時の挙動が明確か。
- 今回やらないことがPR本文から分かるか。

## 実装

- 命名から意図が読み取れるか。
- 責務が大きすぎないか。
- 不要な重複や過剰な抽象化がないか。
- 既存の設計や書き方と大きくずれていないか。
- 例外処理やログが必要な場所にあるか。

## テスト

- 正常系だけでなく異常系も確認しているか。
- 境界値を確認しているか。
- 既存機能への影響を確認しているか。
- 手動確認だけで十分か、自動テストも必要か。

## GitHub

- PRの目的が本文から分かるか。
- 差分が大きすぎないか。
- CIが通っているか。
- 不要なファイルやデバッグ用の差分が混ざっていないか。
- Labels、Milestone、関連Issueが必要に応じて設定されているか。

## コメントの書き方

- 事実、理由、提案を分けて書く。
- 好みだけの指摘なら、その前提を明記する。
- 直してほしいものと、任意の提案を区別する。
- 大きな設計相談は、1行コメントではなくPR全体のコメントにまとめる。

```md
この分岐だと空配列のときに0件表示にならず、前回の結果が残りそうです。
`items.length === 0` の場合に結果を空にする処理を追加するとよさそうです。
```
